## 0. Cadrage

### 0.1 Objectif

Audit produit, fonctionnel et production-ready de SmartVest, **page par page** + **transverse** (onboarding, brokers, banking, doc, sécurité). Livrer une checklist Go Live exploitable par une équipe design + dev + QA + ops.

### 0.2 Niveau cible

- **Bêta privée fermée** par défaut (`SAFE_PUBLIC_MODE=true`, simulation only, 5-50 users invités)
- **Ambition prod publique court terme** : audit conduit comme si on visait (c), mais distinction claire entre exigences bêta vs prod.

### 0.3 Périmètre

**INCLUS** : frontend (UX/parcours/états), backend (logique métier, garde-fous), DB (schéma, RLS), compliance (wording, MIFID, audit chain, RGPD).

**EXCLUS** : infra/devops, tests CI/CD (signalés si trous évidents).

### 0.4 Benchmarks

- **Trade Republic** — UX grand public, simplicité bancaire
- **Composer** — no-code/low-code quant, IA pilotée
- **TradingView** — cockpit analyse profond
- *Filigrane* : IBKR/Saxo pour exécution/connectivité

### 0.5 Méthodologie d'évaluation (grille 9 dimensions)

| Dim | Question |
|---|---|
| 1. Objectif métier | Pourquoi cet onglet existe ? |
| 2. Inputs | Quelles données entrent ? |

### 2.1 `/lisa` — Cockpit AI Analyst (1413 lignes)

**Verdict** : 🟡 Partiel — bêta viable, prod fragile. Architecture cockpit solide, fragile en compliance.

**3 bloquants prod public** :

1. **Feature-flag gate manquant** — la checkbox "Auto-approbation" est exposée sans lire `NEXT_PUBLIC_FEATURE_DELEGATION_HYBRID_SUGGESTIVE`. Un user `SAFE_PUBLIC_MODE` peut activer l'autopilot sans gating. Viole CLAUDE.md section 3.

2. **Wording auto-approve dialog** — la première ligne du dialog dit "Lisa ouvrira et fermera des positions toute seule" sans expliciter `SIMULATION PAPER UNIQUEMENT`. Le disclaimer existe ligne 286 mais buried. Viole CLAUDE.md section 1 (positionnement).

3. **Pas d'error boundary** — un timeout Claude API ou un crash composant cassent la page silencieusement. Aucun fallback UX.

**Gaps importants (bêta)** :

- Pas d'audit UI des approbations user (timeline visible) → non-repudiation faible
- HARVEST cascade UI sparse (DailyHarvestTracker ≠ DailyHarvestPanel scope)
- VIX/DXY/Funding rate pas affichés en mini-panel — user blind aux triggers events
- `NextCycleCountdown` flou : "au plus tard dans N min" sans expliciter événements possibles
- Drawdown hard-kill silencieux côté UI alors que backend déclenche le kill-switch

**Top 3 recos prioritaires** :

| # | Action | Effort | Criticité |
|---|---|---|---|
| 1 | Lire `feature-flags.ts` + gater la checkbox auto-approve | 2h | 🔴 BLOQUANT prod |
| 2 | Réécrire dialog confirmation : `SIMULATION PAPER UNIQUEMENT` en H1 | 1h | 🔴 BLOQUANT prod |
| 3 | Wrap page dans `<ErrorBoundary>` + UI offline-aware | 3h | 🟠 IMPORTANT bêta |

**Microcopy proposée pour reco #2** :

```
ACTIVATION DU MODE AUTONOME — SIMULATION PAPER UNIQUEMENT

⚠️ CECI N'EST PAS DU TRADING RÉEL. Lisa gère des positions VIRTUELLES
dans une simulation sans argent réel.

• Lisa scannera toutes les 7 min et SIMULERA des ouvertures/fermetures
• Aucun ordre réel ne sera jamais envoyé vers un broker
• Tu peux arrêter à tout moment : bouton "Stop immédiat" reste actif

Durée : jusqu'à [date+heure expiration auto]
Confirmer ?
```

### 2.2 `/settings/delegation` — Mandats AutonomyMandate (453 lignes)

**Verdict** : 🟡 Partiel — fondation correcte (architecture, audit chain, RLS), 5 kills critiques.

**5 bloquants bêta** :

1. **Validation Zod trop lâche** — front accepte `maxPositionSizePct` jusqu'à 100 % alors que backend enforce 50 %. User saisit 60 %, soumet, reçoit erreur backend. UX cassée. **Fix : aligner Zod = backend caps (50/50/30/25)**.

2. **Pas de countdown expiration** — la date `expiresAt` s'affiche en dur, pas en "J-15" / "EXPIRÉ". User oublie l'échéance, le mandat se révoque silencieusement. **Fix : composant `<TimeToExpiry>` avec couleur amber (J-7) / red (J-0)**.

3. **Pas de transition auto `expired`** — un mandat qui dépasse `expiresAt` reste `active` en DB jusqu'à la prochaine action. Race condition possible : exécution autonome sur mandat techniquement expiré. **Fix backend : cron job qui flag `expired` et écrit audit `mandate_expired`**.

4. **Pas de confirmation avant activation** — clic "Activer" déclenche immédiatement sans relire les limites. **Fix : modal "Activer ce mandat ?" avec récap des limites**.

5. **Champs `maxSingleTradeNotional` exposés en API/Zod mais non en form** — incohérence produit, futur bug.

**Conformité CLAUDE.md section 6 — checklist** :

| Champ obligatoire | Présent UI | Validation |
|---|---|---|
| `maxPositionSizePct` | ✅ | ⚠️ Zod max 100 vs backend 50 |
| `maxSingleTradePct` | ✅ | ⚠️ Idem |
| `maxDailyTradePct` | ✅ | ⚠️ Idem |
| `allowedAssetClasses` | ✅ | ✅ |
| `forbiddenTickers` | ✅ | ✅ |
| `requiresHumanAbovePct` | ✅ | ✅ |
| `stopLossTriggerPct` | ✅ | ✅ |
| `expiresAt` | ✅ | ✅ obligatoire, max 1 an |
| `killSwitchActive` | ✅ | ✅ jamais gaté par flag |

**Conforme structurellement** — kill-switch toujours visible ✅, pas de mandat permanent ✅, audit hash-chaîné ✅. Les kills sont UX/cohérence client/serveur, pas architecturaux.

### 2.3 `/settings/hyper-trading` — Mode HYPER_ACTIVE (421 lignes)

**Verdict** : 🟢 ~87 % prod-ready. La meilleure page du lot. Conformité CLAUDE.md 6 bis quasi totale.

**Conformité CLAUDE.md section 6 bis — checklist** :

| Règle immuable | Statut |
|---|---|
| Strictement opt-in (pas de profil par défaut) | ✅ |
| Statut `draft` à la création, activation explicite | ✅ |
| Renforce — ne relâche jamais les garde-fous | ✅ (PolicyEngine côté backend) |
| Kill-switch JAMAIS gaté par feature flag | ✅ |
| Kill-switch confirmation 2-step + raison obligatoire | ✅ |
| `expiresAt` obligatoire, vérifié runtime | ✅ |
| Audit hash-chaîné SHA256 prev_hash | ✅ |
| Wording interdit absent ("turbo", "booster", "battre marché") | ✅ |
| `HyperTradingPolicyEngine` retourne `allow/block/require_review/kill_switch` jamais `execute` | ✅ |

**3 gaps UX (non-bloquants prod, importants pour bêta)** :

1. **Les 4 auto-kill-switches sont cachés** — `killSwitchOnAbnormalLoss`, `killSwitchOnDataProviderFailure`, `killSwitchOnBrokerSyncMismatch`, `killSwitchOnVolatilityShock` ne sont **jamais affichés en UI**. User ignore si son profil peut auto-tuer. **Fix : section dédiée "Conditions d'auto-suspension" en lecture seule avec badges**.

2. **Pas de countdown `expiresAt`** — date affichée en dur, pas de "expire dans 5j 12h". **Fix : warning amber si < 7j, red si expiré**.

3. **Pas d'édition des guardrails** — hook `useUpdateGuardrail` existe mais aucun bouton UI. Config figée à la création. **Fix : bouton "Éditer" + modal qui réinjecte les valeurs**.

**Top reco** : exposer les 4 auto-kill-switches dans un encart `bg-red-50` avec badges `AUTO-KILL ACTIVÉ` / `Inactif`. Microcopy : *"Le moteur suspend immédiatement l'évaluation si l'une de ces conditions est détectée."*

### 2.4 `/onboarding` — Wizard 8 étapes

**Verdict** : 🟢 ~85 % prod-ready. Wizard propre, disclaimer obligatoire, calcul profil de risque correct.

**Étapes implémentées** :

```
welcome → disclaimer → currency → horizon → tolerance → goal → profile → portfolio
```

**Conformité MIFID/wording** :

- ✅ Disclaimer obligatoire bloquant (checkbox `J'ai lu et compris...`)
- ✅ Wording correct : *"SmartVest est un outil personnel d'analyse et de simulation. Il ne fournit pas de conseil en investissement personnalisé au sens réglementaire."*
- ✅ *"Les performances passées ne préjugent pas des performances futures"* présent
- ✅ Renvoi explicite vers professionnel réglementé : *"consultez un professionnel réglementé (conseiller financier, CIF, etc.)"*
- ✅ Étape Tolerance ajoute : *"Cette réponse est déclarative et module les paramètres des simulations. Elle ne constitue pas un profil réglementaire."*

**Gaps détectés** :

1. **Bloquant bêta vs prod** : pas de **connexion broker dans le flow d'onboarding**. Le wizard se termine sur la création d'un portfolio vide. Le user externe ne sait pas comment importer ses positions. **Fix : ajouter étape `import-or-skip` après `portfolio` avec deux options : (a) importer CSV broker, (b) saisie manuelle, (c) commencer à zéro**.
2. **Pas de KYC** — par design en `SAFE_PUBLIC_MODE` (simulation only). À ajouter explicitement quand on passe à l'exécution réelle (CLAUDE.md section 9).
3. **Pas de devise FX** — la base currency est demandée (EUR/USD/GBP probablement) mais pas de conversion automatique pour positions multi-devises.
4. **Pas de fuseau horaire** — défaut implicite Europe/Paris ? À rendre explicite (impact sur `sessionStartTime/End` HARVEST).
5. **Pas de fiscalité** — quel régime fiscal (PEA, CTO, IRA, etc.) ? Hors scope MIFID mais important pour reporting.
6. **Pas de recap final** — l'écran portfolio n'affiche pas un récap des 7 étapes précédentes pour relecture avant submit.
7. **Pas de tour guidé post-onboarding** — l'utilisateur arrive sur `/` (dashboard) sans tooltip / coach mark / first-run experience.

**Top 3 recos prioritaires** :

| # | Action | Effort | Criticité |
|---|---|---|---|
| 1 | Ajouter étape `import-or-skip` (CSV broker / manuel / zéro) | 1j | 🟠 IMPORTANT bêta |
| 2 | Ajouter étape récap final avant submit | 4h | 🟠 IMPORTANT bêta |
| 3 | First-run experience post-onboarding (intro.js, 5 spots clés) | 2j | 🟡 NICE-TO-HAVE |
